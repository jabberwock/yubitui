  <action>
    Add new programmatic functions to src/yubikey/pin_operations.rs. Keep the existing interactive functions but mark them `#[deprecated]` (they will be removed in Plan 04 audit). Per D-01: use `--pinentry-mode loopback --status-fd 2 --command-fd 0`. Per D-02: write PINs to gpg's stdin when status-fd signals `GET_HIDDEN`.

    Note on fd choice: use `--status-fd 2` (stderr) so that stdout remains available for normal gpg output. The subprocess captures stderr via `Stdio::piped()` and parses it for `[GNUPG:]` lines.

    1. **Helper function** `fn run_gpg_pin_operation(card_edit_commands: &[&str], pins: &[&str]) -> Result<PinOperationResult>`:
       - Spawn `gpg --card-edit --pinentry-mode loopback --status-fd 2 --command-fd 0` with stdin=piped, stdout=piped, stderr=piped
       - Write card-edit commands to stdin (e.g., "admin\npasswd\n1\nq\n")
       - Read stderr line-by-line in a loop. For each line:
         - Call `gpg_status::parse_status_line(line)`
         - If `GetHidden { .. }`: write next PIN from `pins` iterator to stdin, followed by newline
         - If `Error { .. }` or `ScOpFailure(..)`: record in result as failure
         - If `ScOpSuccess` or `GotIt` after PIN write: record as success step
         - Translate all statuses to human messages via `gpg_status::status_to_message()`
       - Wait for child process exit
       - Return `PinOperationResult { success: bool, messages: Vec<String> }`

    2. `PinOperationResult` struct:
       ```rust
       pub struct PinOperationResult {
           pub success: bool,
           pub messages: Vec<String>,
       }
       ```

    3. **Programmatic functions** (each calls `run_gpg_pin_operation` with appropriate commands and PINs):

       `pub fn change_user_pin_programmatic(current_pin: &str, new_pin: &str) -> Result<PinOperationResult>`:
       - commands: `["admin", "passwd", "1", "q"]`
       - pins: `[current_pin, new_pin, new_pin]` (current, new, confirm)

       `pub fn change_admin_pin_programmatic(current_pin: &str, new_pin: &str) -> Result<PinOperationResult>`:
       - commands: `["admin", "passwd", "3", "q"]`
       - pins: `[current_pin, new_pin, new_pin]`

       `pub fn set_reset_code_programmatic(admin_pin: &str, reset_code: &str) -> Result<PinOperationResult>`:
       - commands: `["admin", "passwd", "4", "q"]`
       - pins: `[admin_pin, reset_code, reset_code]`

       `pub fn unblock_user_pin_programmatic(reset_code_or_admin: &str, new_pin: &str) -> Result<PinOperationResult>`:
       - commands: `["admin", "passwd", "2", "q"]`
       - pins: `[reset_code_or_admin, new_pin, new_pin]`

    4. Important implementation details:
       - Use `std::io::BufReader` to read stderr line-by-line
       - Write all card-edit commands to stdin first (they are queued), then handle PIN prompts as they come from status-fd
       - Actually, gpg in card-edit mode processes commands sequentially. The pattern is: write command, read status response, write PIN when prompted. Use a thread or blocking reads since stdin/stderr need concurrent access. Simplest approach: write all commands to stdin, close command part, then write PINs in response to GET_HIDDEN prompts. Since `--command-fd 0` shares stdin with PIN entry, sequence matters.
       - Better approach: use a single stdin pipe. Write card-edit commands first. Then read status-fd. When GET_HIDDEN appears, write the next PIN. This requires reading stderr in a thread while writing to stdin in the main thread.
       - Use `std::thread::spawn` to read stderr in a background thread, collecting status lines into a `Vec<String>`. Main thread writes commands, then PINs as needed based on a channel or shared state.
       - SIMPLEST correct approach: Write all commands to stdin ("admin\npasswd\n1\nq\n"), then for each GET_HIDDEN prompt, write the corresponding PIN. Use `BufReader::new(child.stderr.take().unwrap())` and iterate lines. When a line matches GET_HIDDEN, write PIN to stdin. This works because gpg buffers commands and processes them sequentially.

    5. Mark old functions with `#[deprecated(note = "Use _programmatic variant instead")]` but keep them for now.

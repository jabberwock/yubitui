## Spacing Scale

Terminal UIs measure spacing in character cells (columns/rows), not pixels.
The 8-point pixel scale maps to a 2-cell unit scale for terminal grids.

| Token | Cell Value | CSS Equivalent | Usage |
|-------|-----------|----------------|-------|
| xs | 1 cell | `padding: 0 1` | Icon-label gaps, tight inline padding |
| sm | 1 cell vertical | `padding: 1` | Button internal padding (height = content + 2) |
| md | 2 cells | `padding: 1 2` | Default widget inner padding |
| lg | 1 row | `margin-bottom: 1` | Section separation within a screen |
| xl | 2 rows | `margin-bottom: 2` | Major section breaks (header/content/footer) |

Exceptions:
- Button minimum height: 3 rows (1 content row + 1 top border + 1 bottom border). Source: RESEARCH.md TCSS example `height: 3`.
- Button minimum width: 16 columns. Source: RESEARCH.md TCSS example `min-width: 16`.
- Footer height: fixed at 1 row (textual-rs Footer renders in a single cell row).
- Header height: fixed at 1-3 rows depending on theme (textual-rs Header default).
- Sidebar width: exactly 33% of terminal width (rule-of-thirds). Source: CONTEXT.md D-13.

## Typography

Terminal rendering is character-cell based ŌĆö no font size control. "Size" here means
visual hierarchy achieved through ratatui Modifiers, content width, and placement.

| Role | Modifier | Line Height | Usage |
|------|----------|-------------|-------|
| Screen title (Header) | BOLD | 1 row | Per-screen title in Header widget. e.g. "Keys", "PIN Management" |
| Section heading | BOLD | 1 row | Block borders with `.title("...")` labels |
| Body | NORMAL (none) | 1 row per line, 1 blank row between sections | Status text, list items, descriptive copy |
| Muted / secondary | DIM | 1 row | Help text, hints, secondary labels |
| Error / warning | BOLD + theme `$error` color | 1 row | Inline error messages |
| Accent label | theme `$accent` fg only, no modifier | 1 row | Keybinding labels in Footer, active selection highlight |

**Weights declared: 2** ŌĆö NORMAL (body, muted) and BOLD (headings, errors).
**Line height:** 1 character row per line; blank separator row (1 row) between logical groups.

## Color

Terminal colors are controlled by textual-rs TCSS theme variables, not hex values.
Hardcoded `Color::` values are explicitly forbidden by RESEARCH.md "Don't Hand-Roll" table.
All colors resolve from the active theme (default: tokyo-night).

| Role | TCSS Variable | 60/30/10 | Usage |
|------|--------------|----------|-------|
| Dominant surface (60%) | `$background` | 60% | Screen background ŌĆö fills all unstyled areas |
| Secondary surface (30%) | `$surface` | 30% | Block/panel backgrounds: sidebar, status panels, popup overlays |
| Accent (10%) | `$accent` | 10% | Reserved-for list below ŌĆö never used for general decoration |
| Success semantic | `$success` | semantic | Healthy status indicators (PIN retries OK, key present) |
| Warning semantic | `$warning` | semantic | Needs-attention states (low retries, unset keys) |
| Error / destructive | `$error` | semantic | Destructive confirmation dialogs, factory reset, error messages |
| Focus ring | `$accent` border | UI state | Button:focus border only |
| Primary text | `$text` | text | All body copy |
| Muted text | `$text-muted` | text | Secondary labels, hints, dim copy |

**Accent (`$accent`) reserved for:**
1. Focused Button border (`Button:focus { border: tall $accent; }`)
2. Footer keybinding key labels (the key character itself, e.g. `[Q]`)
3. Active navigation item highlight in sidebar list
4. Selected menu item background in context menus

**Accent is NOT used for:** general borders, section titles, status icons, decorative elements.

**Existing screen color patterns being migrated (for reference, not hard constraints):**
- Title bar: was `Color::Cyan BOLD` ŌĆö maps to `$accent` or `$primary` in theme
- Status border: was `Color::Green` ŌĆö maps to `$success`
- Warning: was `Color::Yellow` ŌĆö maps to `$warning`
- Error: was `Color::Red` ŌĆö maps to `$error`

## Layout Contract

Source: CONTEXT.md D-13 (rule-of-thirds), D-14 (visually obvious mouse regions).

### Universal Screen Structure

Every screen follows this vertical composition:

```
+------------------------------------------+
| Header (screen title, 1-3 rows)          |
+------------------------------------------+
| Content area (flex-grow: 1)              |
|  +--sidebar 33%--+----main 67%----------+|
|  | navigation /  | primary content       ||
|  | summary info  | (list, form, detail)  ||
|  +--------------++-----------------------+|
+------------------------------------------+
| Footer (keybindings, 1 row, always)      |
+------------------------------------------+
```

Screens without a sidebar (Help, simple info screens) use full-width main content.

### TCSS Baseline (all screens inherit)

```css
Screen {
    layout: vertical;
}

#content {
    layout: horizontal;
    height: 1fr;
}

#sidebar {
    width: 33%;
    min-width: 20;
}

#main {
    width: 1fr;
}

Button {
    border: inner;
    min-width: 16;
    height: 3;
}

Button:focus {
    border: tall $accent;
}

Button:hover {
    background: $surface;
}
```

### Per-Screen Layout Notes

| Screen | Sidebar | Main Content |
|--------|---------|--------------|
| Dashboard | Device list + quick status | Navigation buttons for all 6 screens |
| Keys | Key slot summary (SIG/ENC/AUTH status) | Active operation / wizard steps |
| PIN Management | PIN type selector (User/Admin/Reset) | PIN input form / operation result |
| SSH | Agent status summary | SSH key details / action buttons |
| Diagnostics | Category list | Diagnostic detail / result |
| PIV | Slot list (4 PIV slots) | Slot detail / certificate info |
| Help | Topic list | Topic content (scrollable) |

## Interaction Contract

Source: CONTEXT.md D-05 (Button as click primitive), D-06 (all navigable elements are Buttons),
D-07 (Footer keybindings always visible), D-09 (Pilot-based tests).

### Click Target Rules

1. Every navigation destination is a `Button` widget ŌĆö no raw `Paragraph` click regions.
2. Every destructive action is a `Button` with `$error` border styling.
3. Buttons have minimum 3-row height and 16-column width ŌĆö no narrow unclickable targets.
4. `ClickRegion` / `ClickRegionMap` infrastructure is retired. Zero manual hit-test math.

### Keyboard Binding Rules

1. Every screen declares all its bindings in `key_bindings()` ŌĆö Footer auto-renders them.
2. No keybinding hint text in Paragraph widgets. The Footer is the single source of truth.
3. Bindings follow existing key assignments (no regressions): `q`/Esc=back, `r`=refresh,
   `1`-`6`=screen navigate, `Tab`=switch device, `?`=help.
4. New binding added this phase: `Ctrl+T` = cycle theme (saves to config on change).

### Focus Model

- Tab key cycles focus between interactive Button widgets on the current screen.
- Enter activates the focused Button.
- Esc navigates back (pop screen or return to Dashboard from any screen).
- Arrow keys scroll within List widgets.

### Modal / Overlay Rules

- PIN input dialogs, key generation wizard steps, confirmation dialogs = pushed screen via
  `ctx.push_screen_deferred()`, not z-ordered overlays on the same screen.
- Factory reset confirmation = pushed screen with prominent `$error` color, explicit
  "Type CONFIRM to proceed" text input requirement.

Source: Existing codebase copy patterns (dashboard.rs, pin.rs) + semantic inference from
REQUIREMENTS.md success criteria.

| Element | Copy | Notes |
|---------|------|-------|
| App title (Header) | `yubitui ŌĆö YubiKey Management` | Replaces emoji prefix from old title bar |
| Dashboard screen subtitle | `Device ready` or `No device` depending on state | |
| Primary CTA (Dashboard) | `Open Keys` | Button label for navigating to Keys screen |
| Primary CTA (PIN screen) | `Change User PIN` | First and most common action |
| Primary CTA (SSH screen) | `Add to Agent` | Primary action in SSH workflow |
| Primary CTA (PIV screen) | `View Slot` | Opens slot detail |

| Empty state body (no YubiKey) | `Insert your YubiKey and press R to refresh. Check the USB connection or run Diagnostics.` | Source: dashboard.rs, adapted ŌĆö no emoji |
| Empty state (no OpenPGP keys) | `No keys configured` | Keys screen, all 3 slots empty |
| Empty state body (no keys) | `Generate or import keys using the buttons below.` | |
| Error state (card busy) | `Card busy ŌĆö another process is using the YubiKey. Close other applications and press R to retry.` | |
| Error state (operation failed) | `Operation failed: {reason}. Press Esc to cancel or try again.` | `{reason}` filled from error type |
| Destructive: factory reset warning | `Factory Reset ŌĆö This will permanently erase all OpenPGP keys and data. This cannot be undone.` | Header copy for reset confirmation screen |
| Destructive: factory reset confirm button | `Reset Device` | $error styled Button |
| Destructive: factory reset cancel button | `Cancel` | Default styled Button, auto-focused |
| Destructive: key delete (PIV slot) | `Delete Slot ŌĆö This will erase the key and certificate in this slot.` | |
| Destructive: key delete confirm | `Delete Slot` | $error styled Button |
| Theme cycle notification | `Theme: {name}` | 1-second transient status in Footer, no modal |
| No hardware on Windows (FIDO2) | Not in scope for Phase 8 | Phase 10 |

**Destructive action pattern:** All destructive actions use pushed screen (not inline confirm).
Cancel is the auto-focused default. Confirm (`$error` Button) requires deliberate tab or click.

## Test Contract

Source: CONTEXT.md D-08, D-09, D-10; RESEARCH.md Pattern 4, Pattern 5.

Every migrated screen must have:
1. **One Pilot smoke test** ŌĆö `TestApp::new(80, 24, ...)` + `pilot.settle().await` + `insta::assert_display_snapshot!`
2. **One Pilot navigation test** ŌĆö `pilot.press(KeyCode::...)` + `pilot.settle()` + buffer assertion
3. **Snapshot re-acceptance step** ŌĆö `cargo insta review` after each screen migration (layout changes invalidate old snapshots)
4. **No tmux tests** ŌĆö `tests/e2e/*.sh` deleted; zero new shell-based tests

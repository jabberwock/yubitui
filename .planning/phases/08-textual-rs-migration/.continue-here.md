## This session (post-phase-08 bug fixes):

- **NavButton widget** (`src/tui/widgets/nav_button.rs`) — new wrapper around Button that
  re-emits `messages::Activated { action }` when pressed. Solves the fundamental problem
  that all Buttons emit the same generic `messages::Pressed` with no identity info,
  making it impossible for a parent to know which button was pressed.

- **DashboardScreen updated** — replaced 6 raw `Button::new(...)` with
  `NavButton::new(label, "nav_N")`. Added `on_event` to handle `NavButton::Activated`
  (calls `on_action`) and raw `KeyEvent`s (checks `key_bindings()`).

- **textual-rs vendored and patched** (`vendor/textual-rs/`) — two framework bugs fixed:
  1. `widget/mod.rs`: Default `on_event` now auto-dispatches bubbled `KeyEvent`s against
     `key_bindings()` → `on_action`. Without this, ALL screen-level keyboard shortcuts
     were silently dropped (Escape/back, number shortcuts, etc.).
  2. `app.rs` (both main loop and TestApp): When `focused_widget` is None (screens with
     no focusable children, e.g. HelpScreen with only Labels), falls back to dispatching
     key events to the screen root. Without this, labels-only screens received zero key events.

- `Cargo.toml` switched from git dep to `path = "vendor/textual-rs/crates/textual-rs"`.
</completed_work>

<context>
Two distinct bugs in the textual-rs event dispatch system:

Bug 1 (NavButton): textual-rs routes key events to the *focused* widget's key_bindings
only. Screens are never focused (can_focus()=false). So screen-level bindings were dead.
The fix makes the default on_event check key_bindings when a bubbled KeyEvent arrives.

Bug 2 (no-focus dispatch): screens with no focusable children (only Labels) left
focused_widget=None, causing the dispatcher to skip dispatch entirely. The fix falls back
to the screen root widget when nothing is focused.

The previous session (before /clear) had hypothesized a WidgetId-tracking approach that
turned out to be incorrect (on_event has no sender ID). The NavButton approach was chosen
instead based on reading the actual textual-rs source.
</context>

- Confirmation popups have Cancel (default focus) and Confirm ($error for destructive) buttons
- Per UI-SPEC: Cancel is auto-focused default; Confirm requires deliberate tab/click

1. **Create** `PinScreen` struct implementing `Widget`:
   - Fields: `pin_state: Reactive<PinState>`, `app_state` reference
   - `compose()`: Header("PIN Management") + sidebar (PIN type selector as Button list) + main (active operation content) + Footer
   - `key_bindings()` per UI-SPEC:
     - Esc=back, c=change_user_pin, a=change_admin_pin, r=set_reset_code, u=unblock_pin
   - `on_action()`: dispatch each action

2. **Sub-screens as pushed screens** (per UI-SPEC Modal/Overlay Rules):
   The current pin.rs has PinScreen enum: Main, ChangeUserPin, ChangeAdminPin, SetResetCode, UnblockPin, FactoryReset. Each non-Main variant becomes a pushed screen:

   - `ChangeUserPinScreen` — pushed via `ctx.push_screen_deferred()`:
     - Shows current PIN input (PinInput widget), new PIN input, confirm new PIN
     - Submit button executes PIN change via model layer
     - Cancel/Esc pops back

   - `ChangeAdminPinScreen` — same pattern as ChangeUserPin
   - `SetResetCodeScreen` — same pattern
   - `UnblockPinScreen` — same pattern (requires reset code + new PIN)

   Each sub-screen is a separate struct implementing Widget, defined within pin.rs (or extracted to sub-modules if too large).

3. **Port rendering logic:**
   - PIN retry count display (from model::pin data)
   - Status indicators using $success/$warning/$error theme variables
   - Primary CTA: "Change User PIN" button (per UI-SPEC copywriting)
   - Destructive actions: use $error-styled Button for factory reset, per UI-SPEC destructive pattern

4. **Delete** old `render()` and `handle_key()` free functions.

5. Keep all model-layer PIN operation calls (pin_operations::change_user_pin etc.) — they are called from on_action(), not changed.

    **Design decision from NOTES.md:** Sequential wizard using pushed screens (matches PinManagementScreen pattern).

    1. **Create AddAccountScreen Widget** in src/tui/oath.rs (below OathScreen):

    ```rust
    #[derive(Default, Clone, PartialEq)]
    pub struct AddAccountState {
        pub step: AddAccountStep,
        pub issuer: String,
        pub account_name: String,
        pub secret_b32: String,
        pub oath_type: OathType,  // default Totp
        pub error_message: Option<String>,
    }

    #[derive(Default, Clone, Copy, PartialEq)]
    pub enum AddAccountStep {
        #[default]
        Issuer,       // Step 1: Enter issuer (e.g., "GitHub")
        AccountName,  // Step 2: Enter account name (e.g., "user@github.com")
        Secret,       // Step 3: Enter Base32 secret
        TypeSelect,   // Step 4: Select TOTP or HOTP (default TOTP)
        Confirm,      // Step 5: Review and confirm
    }
    ```

    **AddAccountScreen Widget struct:**
    ```rust
    pub struct AddAccountScreen {
        state: RefCell<AddAccountState>,
        input_buffer: RefCell<String>,
    }
    ```

    **compose():**
    - Header("Add OATH Account")
    - Step indicator Label: `"Step {N}/5: {step_name}"`
    - Per step:
      - Issuer: Label("Enter issuer name (e.g., GitHub, Google):") + Label showing current input_buffer
      - AccountName: Label("Enter account name (e.g., user@example.com):") + Label showing current input_buffer
      - Secret: Label("Enter Base32 secret key:") + Label showing input_buffer (masked as `"*".repeat(len)`)
      - TypeSelect: Label("Select type:") + Label("[T] TOTP (time-based, default)") + Label("[H] HOTP (counter-based)")
      - Confirm: Label with summary of all fields, Label("Press Enter to save, Esc to cancel")
    - If error_message is Some: Label with the error text
    - Footer with step-appropriate bindings

    **on_action() / on_event():**
    - Handle character input: append to input_buffer (for Issuer/AccountName/Secret steps)
    - Backspace: remove last char from input_buffer
    - Enter: advance to next step, transferring input_buffer to the appropriate state field, clear input_buffer
    - On Confirm step Enter:
      - Build credential name as `"{issuer}:{account_name}"` (or just `"{account_name}"` if issuer empty)
      - Call `crate::model::oath::put_credential(name, secret_b32, oath_type, OathAlgorithm::Sha1, 6)`
      - On Ok: push a PopupScreen("Success", "Account added successfully"), then pop self
      - On Err: set error_message to the error string, stay on Confirm step
    - On TypeSelect: 't' -> TOTP, 'h' -> HOTP, Enter -> keep default (TOTP)
    - Esc on any step: pop self (cancel)

    2. **Wire "add_account" action in OathScreen.on_action():**
    ```rust
    "add_account" => {
        ctx.push_screen_deferred(Box::new(AddAccountScreen::new()));
    }
    ```

    3. **Wire "delete_account" action in OathScreen.on_action():**
    - Get the currently selected credential name
    - If no credentials or no selection: no-op
    - Push a ConfirmScreen:
      ```rust
      let name = credential.name.clone();
      let display_name = credential.issuer.as_deref().unwrap_or(&credential.name);
      ctx.push_screen_deferred(Box::new(ModalScreen::new(Box::new(
          ConfirmScreen::new(
              "Delete Credential",
              format!("Permanently delete '{}'?\n\nThis cannot be undone. The credential will be removed from the YubiKey.", display_name),
          )
      ))));
      ```
    - After ConfirmScreen returns (check confirmed flag): call `crate::model::oath::delete_credential(&name)`
    - Show result popup (success or error)

    4. **Password prompt handling (OATH-05):**
    Add to OathScreen compose(): when `oath_state.password_required == true`:
    - Instead of credential list, show:
      ```
      Label("OATH applet is password-protected.")
      Label("Password management is not yet supported (deferred to v2).")
      Label("Use 'ykman oath access change' to remove the password, then retry.")
      ```
    - This is the v1.1 scope — OATH-07 (set/change password) is deferred to v2 per ROADMAP backlog. We inform the user clearly.

    **IMPORTANT:** The AddAccountScreen should handle character input directly via on_event() matching KeyEvent, not via PinInputWidget — the wizard needs multi-field state and step transitions that PinInputWidget's single-field pattern doesn't support.

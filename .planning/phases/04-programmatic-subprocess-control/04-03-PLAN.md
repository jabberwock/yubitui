Replace the interactive key generation and key import flows with non-interactive TUI-driven equivalents. Key generation uses a multi-step wizard UI + gpg --batch --gen-key. Key import uses gpg --edit-key with --command-fd to auto-map subkeys to card slots by capability.

Purpose: Eliminates terminal escapes for the two most complex key operations (per D-03, D-09-D-14). The user never sees raw gpg output; instead, a guided wizard collects parameters and a progress popup shows operation status.

Output: Rewritten key generation and import in key_operations.rs, wizard screens in keys.rs, rewired app.rs.

    3. `pub fn generate_key_batch(params: &KeyGenParams, admin_pin: &str) -> Result<KeyOperationResult>`:
       Per D-11, use `gpg --batch --gen-key` with a generated parameter file.

       - Create a temp file (using `tempfile::NamedTempFile` or manual creation in std::env::temp_dir()) with gpg batch content:
         ```
         %echo Generating OpenPGP key
         Key-Type: {key_type}
         Key-Length: {key_length}     # only for RSA
         Key-Curve: {curve}           # only for Ed25519
         Subkey-Type: {subkey_type}
         Subkey-Length: {subkey_length}  # only for RSA
         Subkey-Curve: {subkey_curve}   # only for cv25519
         Name-Real: {params.name}
         Name-Email: {params.email}
         Expire-Date: {params.expire_date}
         %no-protection
         %commit
         %echo done
         ```
         For Ed25519: Key-Type=EDDSA, Key-Curve=ed25519, Subkey-Type=ECDH, Subkey-Curve=cv25519
         For RSA2048: Key-Type=RSA, Key-Length=2048, Subkey-Type=RSA, Subkey-Length=2048
         For RSA4096: Key-Type=RSA, Key-Length=4096, Subkey-Type=RSA, Subkey-Length=4096

       - Spawn: `gpg --batch --status-fd 2 --gen-key <tempfile_path>`
         stdin=piped, stdout=piped, stderr=piped

       - Read stderr for status lines, translate via gpg_status
       - On KEY_CREATED: record fingerprint in result
       - Wait for exit, clean up temp file
       - Return `KeyOperationResult { success, messages, fingerprint: Option<String> }`

       - Note: `%no-protection` is used because the card PIN is the protection (per D-11). The key will be moved to the card afterward.

       - If params.backup is true: after generation, run `gpg --export-secret-keys --armor --output <backup_path> -- <fingerprint>` to create backup before moving to card. Use `--pinentry-mode loopback` and write admin_pin if prompted.

    **B. Key import backend (per D-12, D-13, D-14):**

    5. `pub fn import_key_programmatic(key_id: &str, admin_pin: &str) -> Result<ImportResult>`:
       Per D-12: use `gpg --edit-key -- <keyid> --pinentry-mode loopback --status-fd 2 --command-fd 0`

       Per D-13: auto-map subkeys by capability. First, discover subkey capabilities:
       - Run `gpg --list-keys --with-colons -- <key_id>` to get subkey records
       - Parse `sub` lines: field 12 contains capability flags (s=sign, e=encrypt, a=auth)
       - Build a plan: for each subkey with capability, map to card slot (1=SIG, 2=ENC, 3=AUT)

       Then for each subkey to move:
       - Write to stdin: `key N\nkeytocard\n{slot_number}\n`
       - When GET_HIDDEN appears (for admin PIN), write admin_pin
       - After all subkeys: write `save\n`

       Per D-14: return `ImportResult`:
       ```rust
       pub struct ImportResult {
           pub sig_filled: bool,
           pub enc_filled: bool,
           pub aut_filled: bool,
           pub messages: Vec<String>,
       }
       ```

       Format result message: "SIG {check_or_dash}  ENC {check_or_dash}  AUT {check_or_dash}"
       where check = unicode checkmark, dash = em-dash

    6. Helper `fn parse_subkey_capabilities(key_id: &str) -> Result<Vec<SubkeyInfo>>`:
       ```rust
       struct SubkeyInfo {
           index: usize,      // 1-based subkey index for gpg "key N" command
           capabilities: String,  // e.g. "e", "s", "a"
       }
       ```
       Parse `gpg --list-keys --with-colons -- <key_id>` output. Count `sub` lines to get index. Extract field 12 for capabilities.

    7. Mark old `generate_key_on_card()` and `import_key_to_card()` with `#[deprecated]`.

    8. Add `tempfile` to Cargo.toml dependencies if not already present (check first). If not available, use `std::fs::write` to a path in `std::env::temp_dir()` with a timestamped filename, and clean up manually.

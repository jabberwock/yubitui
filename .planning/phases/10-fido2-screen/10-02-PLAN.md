- `Char('r')` -> "reset" (description: "R Reset", show: true)
- `Char('p')` -> "authenticate_pin" (description: "P Unlock", show: true)

**Widget impl for Fido2Screen:**

`widget_type_name` returns "Fido2Screen".

`compose()` (per D-06 layout — Header, info section, passkeys section, footer):
- Always push Header::new("FIDO2 / Security Key") per D-06
- Match on self.fido2_state:
  - `None` -> Label "No YubiKey detected. Insert your YubiKey and press Esc to return."
  - `Some(state)` ->
    - **Info section (always visible per D-03, no PIN needed):**
      - Label: "Firmware: {state.firmware_version.as_deref().unwrap_or("Unknown")}"
      - Label: "Algorithms: {state.algorithms.join(", ")}" (or "None reported" if empty)
      - Label for PIN status: if state.pin_is_set { "PIN: Set ({state.pin_retry_count} retries remaining)" } else { "PIN: Not set" }
      - Empty Label spacer
    - **Passkeys section:**
      - If !state.pin_is_set (per D-04):
        - Label: "No PIN configured -- press S to set one."
      - Else if state.credentials.is_none() (locked, per D-05):
        - Label: "Credentials locked -- press P to authenticate"
      - Else if !state.supports_cred_mgmt:
        - Label: "Passkey management requires CTAP 2.1 (not supported by this device)"
      - Else match credentials:
        - Some(creds) if creds.is_empty():
          - Label: "No passkeys stored on this device."
        - Some(creds):
          - Label: "Passkeys ({creds.len()})"
          - For each (idx, cred) in creds.iter().enumerate():
            - marker = if idx == selected_index { ">" } else { " " }
            - Label: "  {marker} {cred.rp_id:<30}  {cred.user_name}"
- Always push empty Label spacer then Footer

`key_bindings()` returns FIDO2_BINDINGS.

`on_event()` — same pattern as OathScreen: downcast KeyEvent, match against key_bindings, call on_action. Also handle character input for PIN entry if a PIN sub-screen is pushed.

`on_action()`:
- "back" -> ctx.pop_screen_deferred()
- "up" -> decrement selected_index if > 0 via state.update()
- "down" -> increment selected_index if < credentials.len()-1 via state.update()
- "set_pin" -> if fido2_state.pin_is_set { push PinChangeScreen } else { push PinSetScreen }
- "delete_credential" -> get selected credential from fido2_state.credentials, if Some push DeleteCredentialScreen with credential info
- "authenticate_pin" -> push PinAuthScreen (prompt for PIN, then enumerate credentials)
- "reset" -> no-op in this plan (wired in Plan 03)

`render()` — empty (compose handles rendering).

**PinSetScreen (pushed screen for FIDO-02 — set new PIN):**
```rust
pub struct PinSetScreen {
    new_pin: RefCell<String>,
    confirm_pin: RefCell<String>,
    step: RefCell<PinSetStep>,  // enum { EnterNew, ConfirmNew }
    error_message: RefCell<Option<String>>,
}
```
- compose(): Header "Set FIDO2 PIN", shows current step prompt with input buffer (masked with *)
- on_event(): handles character input (Char -> push to buffer), Backspace (pop), Enter (advance step), Esc (cancel/pop)
- Step 1 (EnterNew): "Enter new PIN (min 4 characters):" with masked input
- Step 2 (ConfirmNew): "Confirm new PIN:" — on Enter, check match. If match, call crate::model::fido2::set_pin(&new_pin). On success, pop and push PopupScreen "PIN set successfully". On error, show error message.
- PIN minimum length validation: if new_pin.len() < 4, show error "PIN must be at least 4 characters"

**PinAuthScreen (pushed screen for credential unlock):**
```rust
pub struct PinAuthScreen {
    pin_input: RefCell<String>,
    error_message: RefCell<Option<String>>,
}
```
- compose(): Header "Authenticate FIDO2 PIN", Label "Enter FIDO2 PIN:", masked input
- on_event(): Char input, Backspace, Enter (call enumerate_credentials), Esc (cancel)
- On success: pop screen. The parent Fido2Screen will need to be reconstructed with credentials loaded. Since we can't mutate the parent's state from a pushed screen, the PinAuthScreen should call enumerate_credentials and then push a NEW Fido2Screen with the updated state (pop current Fido2Screen + push new one). Implementation: on Enter, call model::fido2::enumerate_credentials(&pin). On success, pop self, then use ctx to communicate the result. The simplest approach: pop self with ctx.pop_screen_deferred(), then the parent Fido2Screen handles the result. BUT since we can't pass data back easily, use a pragmatic approach: PinAuthScreen calls enumerate_credentials, on success pops self and pops the Fido2Screen, then pushes a new Fido2Screen with credentials populated (get fresh fido2_info and merge credentials).
- On error: show error inline "Invalid PIN" or the error message from the crate.

**DeleteCredentialScreen (wraps ConfirmScreen, same pattern as OATH's DeleteConfirmScreen):**
```rust
pub struct DeleteCredentialScreen {
    rp_id: String,
    user_name: String,
    credential_id: Vec<u8>,
    cached_pin: Option<String>,
    inner: ConfirmScreen,
}
```
- Constructor takes rp_id, user_name, credential_id, cached_pin (from Fido2TuiState)
- compose() delegates to inner.compose()
- key_bindings() delegates to inner.key_bindings()
- on_action("confirm"): if cached_pin is Some, call model::fido2::delete_credential(&pin, &credential_id). On success: pop and push PopupScreen "'{rp_id}' credential deleted.". On error: pop and push PopupScreen with error. If cached_pin is None, push PinAuthScreen first (user must authenticate before deleting).
- on_action("cancel"): pop screen

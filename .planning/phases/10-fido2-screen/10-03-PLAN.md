<!-- From research: authenticatorReset protocol -->
CTAP HID Reset Protocol:
1. Open FIDO HID device via hidapi::HidApi
2. CTAPHID_INIT: send 8-byte nonce on broadcast channel (0xFFFFFFFF), receive 4-byte channel ID
3. CTAPHID_CBOR: send authenticatorReset command (0x10 HID cmd, 0x07 CTAP2 cmd byte) on allocated channel
4. Read response: 0x00 = success, 0x2E = CTAP2_ERR_NOT_ALLOWED (outside 10s window)

CTAPHID frame format (from FIDO HID spec):
- Initialization packet: [channel_id(4)] [cmd(1)|0x80] [bcnt_hi(1)] [bcnt_lo(1)] [data(57)]
- CTAPHID_INIT cmd = 0x06, CTAPHID_CBOR cmd = 0x10
- For CTAPHID_INIT: data = 8-byte nonce; response = nonce(8) + new_channel_id(4) + ...
- For authenticatorReset: CBOR payload = single byte 0x07 (command number)

Device detection for replug:
- ctap-hid-fido2 provides FidoKeyHidFactory::create() which returns Ok if device found
- Poll this in a loop to detect device reconnection after unplug

From textual-rs worker API:

```rust
/// CTAP HID constants for raw reset frame
const CTAPHID_INIT: u8 = 0x06;
const CTAPHID_CBOR: u8 = 0x10;
const BROADCAST_CID: [u8; 4] = [0xFF, 0xFF, 0xFF, 0xFF];
const FIDO_USAGE_PAGE: u16 = 0xF1D0;
const FIDO_USAGE: u16 = 0x01;

/// CTAP2 error codes
const CTAP2_OK: u8 = 0x00;
const CTAP2_ERR_NOT_ALLOWED: u8 = 0x2E;
```

3. Add `pub fn find_fido_hid_device_path() -> Result<String>`:
   - Open hidapi::HidApi::new()
   - Iterate device_list(), find first device where usage_page == FIDO_USAGE_PAGE (0xF1D0) and usage == FIDO_USAGE (0x01)
   - Return the device path as String
   - If not found, return error "No FIDO HID device found"

4. Add `pub fn is_fido_device_present() -> bool`:
   - Try find_fido_hid_device_path(), return Ok means present, Err means absent.

5. Add `pub fn reset_fido2() -> Result<()>`:
   - Call find_fido_hid_device_path() to get path
   - Open device via HidApi::new()?.open_path()
   - **CTAPHID_INIT**: Build init packet on broadcast channel:
     - Bytes: [0xFF, 0xFF, 0xFF, 0xFF, 0x80 | CTAPHID_INIT, 0x00, 0x08, nonce(8 random bytes), padding to 64 bytes]
     - Write 64 bytes to device
     - Read 64 bytes response (timeout 1000ms)
     - Extract channel_id from response bytes [15..19] (after 8-byte nonce echo)
   - **CTAPHID_CBOR with authenticatorReset**:
     - Build CBOR packet on allocated channel:
     - Bytes: [channel_id(4), 0x80 | CTAPHID_CBOR, 0x00, 0x01, 0x07, padding to 64 bytes]
     - (bcnt = 1 byte payload, payload = 0x07 = authenticatorReset command)
     - Write 64 bytes to device
     - Read 64 bytes response (timeout 30000ms — reset may involve user presence check)
   - Parse response: first byte after channel_id+cmd+bcnt is the CTAP2 status:
     - response[7] == CTAP2_OK (0x00): return Ok(())
     - response[7] == CTAP2_ERR_NOT_ALLOWED (0x2E): return Err("Reset not allowed — device must be freshly plugged in (within 10 seconds)")
     - Other: return Err with hex error code
   - Use `rand::random::<[u8; 8]>()` for nonce generation (rand is already a dependency via ctap-hid-fido2). If rand is not available, use a fixed nonce [0x01..0x08] — CTAPHID_INIT nonce is not security-critical.

Note: The response parsing offset depends on the CTAPHID packet format. The response to CTAPHID_CBOR has: [cid(4), cmd(1), bcnt_hi(1), bcnt_lo(1), status(1), ...]  so status byte is at index 7. Verify this during implementation by checking actual response bytes if tests fail.

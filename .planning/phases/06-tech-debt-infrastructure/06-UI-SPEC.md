## Scope Note

Phase 6 is a pure infrastructure refactoring phase. **No new UI screens are added.** The design contract governs:

1. The `--mock` mode startup banner (new, one-time display in status bar)
2. The CI lint error message format (developer-facing, not end-user)
3. The refactored per-screen action enums and `handle_key()` functions — which must produce **identical visible output** to the current implementation
4. The `AppState` struct and `src/model/` layer — zero user-visible changes

All existing screens (Dashboard, Diagnostics, Help, Keys, PIN Management, SSH Setup, PIV) must render identically after the architecture migration. This contract enforces visual regression prevention, not new design decisions.

## Spacing Scale

ratatui uses character-cell rows/columns, not CSS pixels. Tokens below map to `Constraint` values already established in the codebase.

| Token | Value | Ratatui Equivalent | Usage |
|-------|-------|--------------------|-------|
| xs | 1 cell | `Constraint::Length(1)` | Divider rows, inline gaps |
| sm | 2 cells | `Constraint::Length(2)` | Compact padding rows |
| md | 3 cells | `Constraint::Length(3)` | Title bars, section headers |
| lg | 5 cells | `Constraint::Length(5)` | Short info panels |
| xl | 10 cells | `Constraint::Length(10)` | Navigation menu blocks |
| content | remaining | `Constraint::Min(10)` | Primary content area |

**Source:** Codebase scan — `src/ui/dashboard.rs` uses `Constraint::Length(3)` for title, `Constraint::Min(10)` for content, `Constraint::Length(10)` for nav. Established across all screens.

Exceptions: none for Phase 6.

## Typography

ratatui does not have font sizes — all text renders in the terminal's monospace font at a single size. "Weight" maps to `Modifier::BOLD`. Two-weight system is already established.

| Role | Modifier | Color | Usage |
|------|----------|-------|-------|
| Heading | `Modifier::BOLD` | `Color::Cyan` | Screen titles, section headers (e.g., "📊 Status") |
| Body | (none) | `Color::White` or terminal default | All paragraph content, list items |
| Label | `Modifier::BOLD` | `Color::Yellow` | Field names, key-hint labels (e.g., "[R] Refresh") |
| Status — OK | (none) | `Color::Green` | Healthy states, active border accents |
| Status — Warn | (none) | `Color::Yellow` | Warning states (DANGER retry level) |
| Status — Error | `Modifier::BOLD` | `Color::Red` | Error states, BLOCKED PIN, destructive confirmations |

**Source:** `src/ui/dashboard.rs`, `src/ui/pin.rs`, `src/ui/mod.rs` — consistent across all existing screens.

## Color

ratatui named colors, not hex. The terminal renders these; actual RGB depends on the user's terminal theme.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | terminal default bg (`Color::Reset`) | Main content area background — full screen |
| Secondary (30%) | `Color::DarkGray` | Status bar at bottom of every screen |
| Accent — navigation (10%) | `Color::Yellow` | Navigation block borders, key-hint labels |
| Accent — heading (10%) | `Color::Cyan` | Screen titles, active menu items, section heading text |
| Semantic — healthy | `Color::Green` | Status block borders when device OK, occupied slot indicators |
| Semantic — warning | `Color::Yellow` | PIN at 1-2 retries (DANGER level), caution messages |
| Semantic — destructive | `Color::Red` + `Modifier::BOLD` | Error messages, BLOCKED PIN state, destructive confirmations |
| Semantic — special action | `Color::Magenta` | Unblock PIN wizard action only |

Accent reserved for:
- `Color::Cyan`: screen title text, active menu item highlight, diagnostics section headings
- `Color::Yellow`: navigation block border, key-hint bar items, warning-level PIN status text

**Source:** `src/ui/dashboard.rs`, `src/ui/pin.rs`, `src/ui/mod.rs` — all existing screens follow this palette.

## Interaction Contract

### Mock Mode (`--mock` flag)

- When `--mock` is active, the status bar displays the mock indicator permanently at `Color::Yellow` background instead of `Color::DarkGray`.
- Mock YubiKey fixture state is pre-populated: FW 5.4.3, serial 12345678, all 3 OpenPGP slots occupied, SSH configured, PINs at 3/3 retries.
- All screens render normally using fixture data — navigation, key display, PIN display all exercise real code paths.
- No modal or popup is shown at launch for mock mode — the status bar indicator is sufficient.

**Source:** CONTEXT.md D-01, D-02, D-03; REQUIREMENTS.md INFRA-01

### Architecture Migration (invisible to user)

- Each screen's `handle_key()` function must return the same navigation transitions as the current monolithic `app.rs` match arm.
- Per-screen typed action enums must not add or remove any keybindings visible in the current key-hint bar.
- All existing popup/dialog widgets (`src/ui/widgets/`) move to `src/tui/widgets/` path-only — zero behavior change.

**Source:** CONTEXT.md D-09; REQUIREMENTS.md INFRA-05

### 50ms Sleep (INFRA-02)

- No user-visible change. The sleep is between scdaemon kill and the first APDU — transparent to all rendering.
- If a card busy error occurs during verification testing, the error copy above is the display string.

**Source:** CONTEXT.md D-12; REQUIREMENTS.md INFRA-02

## Visual Regression Rule

Because Phase 6 is a refactoring phase with no new screens, the primary quality gate is **visual equivalence**, not new design decisions.

After implementation, every existing screen must produce byte-identical ratatui `TestBackend` output compared to the pre-migration baseline:
- Dashboard screen with YubiKey present
- Dashboard screen with no YubiKey (empty state)
- PIN Management screen (all three sub-states: OK, DANGER, BLOCKED)
- Status bar at each screen

Any diff in TestBackend output is a regression, not a design change.

**Source:** REQUIREMENTS.md INFRA-03, INFRA-04; CONTEXT.md (architecture migration section)
